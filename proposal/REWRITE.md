リライト／改善提案
=================

目的
----
可読性・効率性・保守性・テスタビリティを改善し、将来の拡張やCI導入を容易にする。

短期（最小変更）
-----------------
- `gitr` の Markdown フェンス除去（既に対応）と `set -euo pipefail` の導入
- `shellcheck` を CI に追加して静的解析を実施
- `bats-core` テストを追加して基本的な動作を検証（tests/gitr.bats を追加済み）

中期（構造改善）
-----------------
- URL 正規化・パス計算を小さな関数に切り出す（`normalize_url`, `repo_dir_for` 等）
- 入力検証を強化し、部分一致ではなく厳密マッチ（または明示的なパターン）に変更
- エラーを標準化して返り値・ログへ出力

長期（オプション：言語移植）
---------------------------
検討候補：Go / Python

- Go
  - 長所: 単一バイナリ、標準で強力なテスト・モジュール、並列処理の容易さ
  - 短所: 小さなユーティリティとしては実装コストがやや高い

- Python
  - 長所: 迅速なプロトタイピング、豊富なライブラリ
  - 短所: 実行環境依存、配布はpipやスクリプト経由

推奨移行パス（Go を選ぶ場合の骨子）
-------------------------------
1. `cmd/gitr/main.go` を作成し、現在のシェル補完ロジックはまずは呼び出し対象のサブコマンドとして実装
2. `internal/gitr` パッケージに `ListRoots()`, `FindRepo()`, `Clone()` を実装
3. 単体テストを `internal/gitr` に追加（Go の `testing`）
4. CI（GitHub Actions）で `go test` と `golangci-lint` を実行

次のステップ（提案）
-------------------
1. まず `shellcheck` ワークフローを追加
2. `bats` テストをローカルで実行して確認（CI でも実行）
3. 中期改善（関数分離と入力検証）を段階的に適用
4. 必要なら Go への移行スケジュールを別途作成
